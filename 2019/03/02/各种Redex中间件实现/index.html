前言

最近处于React疯狂痴迷状态，今天记录各种Redux中间件的实现

日志(logger)中间件

logger使用

日志中间件可以记录state在更新前后的值, 利用的是store中的applyMiddleware, applyMiddleware主要思想就是把Redux工作流中的dispatch环节封装，返回一个新的store，该store中其他部分没有变化，只是dispatch被新的dispatch覆盖, 我们先来看看是怎么用的👇

这里的applyMiddleware内部主要是对dispacth做了一层封装，并封盖掉store中旧的dispatch，并返回这个新的store, 那它是怎么实现能够获得旧状态和新状态的呢？ 其实是在logger函数中调用next()之前是没有调用store中的dispatch的, 根据昨天redux实现的笔记中, 只有调用dispatch，diapatch内部才会调用reducer才会更新状态(state),而logger函数内部的next就是dispatch函数啦，看它接受了一个action就知道啦, 知道了大体思想, 后面所有的中间件都会使用他，我们来自己写一下吧👇

applyMiddleware实现

我们大体结合logger来解释下，首先applyMiddleware内部通过调用传入的createStore返回一个store，并把store.dispatch传入了middleware也就是logger中，这时返回一个函数dispatch， 它的函数体就是下边这个样子👇

下一步return以后，store里的dispatch就被这个覆盖掉了，当我们调用store的dispatch方法时也是调用的这个, redux实现笔记中可以看到，旧的disaptch的函数体是这样子的👇

可以看出，旧的dispatch一调用，我们的state就更新了，而新的dispatch函数体(我们后边简写为newDis)里是没有更新state语句的，所以在newDis内部是可以拿到旧的state的，那么我们怎么让state更新呢？ 通过调用next方法并把action传入, 这里的next方法就是store.dispatch, 是applyMiddleware内部给他传入的，就是这一句middleware(store.dispatch), 调用next(action)方法其实就是调用了原来的dispatch，这时状态(state)就可以更新了😄

redux-thunk

thunk中间件是为了解决dispatch中的异步问题，在使用thunk之前，我们dispatch的参数只能是一个action对象，如果我们想利用dispatch发起异步请求的话，就需要thunk的帮助了，看下thunk封装后的dispatch是怎么使用的，我们根据使用来写thunk中间件👇

看来封装后的dispatch不是接收一个action了，而是一个回调，那我们来写下吧

我们在上边logger中说到,thunk尾部的action => { ... }这个函数就是我们封装以后的dispatch, 新的dispatch接受一个回调，所以我们先检测action的类型，是函数则直接调用传入的回调，并把原来的dispatch 也就是next传入, 不是函数则是同步调用，直接调用原来的dispatch

promise 和 thunk的作用一致， 都是解决异步问题的，只不过promise写起来更装犊子优雅一点😄， 看写法

中间件的链式调用

不知道各位有没有对中间件的第二个参数next有所疑问，next明明是传入的dispatch，为什么叫next，在自己练习的过程中也估计把next写成dispatch，这样好理解一点，终于学到中间件最后，谜底揭晓了, 其实next是为中间件的链式调用准备的, 什么是链式调用呢，我们再拿开头的那个logger中间件举个例子

看完例子我们再小小总结一下, 其实就是多个中间件一个个调用，形成一个调用链, 我们笔记一开头写的applyMiddleware 为了实现链式调用就需要升级一丢丢了👇

那我们调用dispatch的顺序又是啥呢？ 我们看这张图分析下
当我们调用dispatch的时候，会先打印logger1更新前, 然后调用next，next就是logger2最内层的action, 然后这时候会调用logger2更新前, 又遇到了logger2中的next， logger2中的next我们上边说了，传入了原始的dispatch, 调用以后就会更新状态，接着logger2更新后, 接着打印logger1更新后
